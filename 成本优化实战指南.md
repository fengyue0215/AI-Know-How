# AI 编程成本优化实战指南

> C++ 项目代码量大、编译单元多、上下文长，token 消耗通常是 Web 开发的 3-5 倍。  
> 本指南基于 2026 年多个工程团队的实战经验整理，教你系统性地控制成本。

---

## 📊 一、为什么 C++ 项目的 AI 成本更高

| 因素 | C++ 项目特点 | Token 影响 |
|------|-------------|-----------|
| 头文件 | 动辄数百行模板/宏/类型定义 | 单文件读取 = 3K-10K token |
| 构建系统 | CMake/Bazel 配置复杂 | 每次上下文包含构建文件 |
| 依赖图 | .h 和 .cpp 分离，跨文件引用多 | 理解一个函数需要读 3-5 个文件 |
| 编译错误 | 模板错误信息极长（数百行） | 错误反馈消耗大量 token |
| 内存模型 | 需要讨论 ownership/lifetime | 更多分析 → 更多输出 token |

> 实测数据：一个中等复杂度的 C++ 功能开发（如"实现 Redis 连接池"），Claude Code 单次会话消耗 50K-200K token，约 $2-8（API 价格）。

## 💰 二、五大省钱杠杆

### 杠杆 1：Prompt Cache —— 成本控制的基石（节省 35-55%）

**原理**：Anthropic API 在每次对话轮次中会重发全部历史。Prompt Caching 将稳定前缀缓存，缓存命中时输入 token 享受 **90% 折扣**（只收 10%）。

```
┌─────────────────────────────────────────────┐
│  API 调用 #1：                               │
│  [系统 Prompt][CLAUDE.md][工具定义][用户消息1] │
│  → 缓存写入：1.25x 价格写                    │
│                                             │
│  API 调用 #2：                               │
│  [系统 Prompt][CLAUDE.md][工具定义] ← 缓存命中 │
│  [助手回复1][用户消息2] ← 增量部分，原价       │
│  → 前缀部分只收 10%！                        │
└─────────────────────────────────────────────┘
```

**5 个最常见的 Cache 破坏陷阱**：

| 陷阱 | 原因 | 后果 | 避免方法 |
|------|------|------|---------|
| ❌ 会话中途编辑 CLAUDE.md | 改变系统 Prompt 前缀 | 缓存全部失效 | 会话开始前一次性写好 |
| ❌ 中途切换模型 | 不同模型有独立的缓存 | 新模型缓存从零开始 | 一次会话坚持一个模型 |
| ❌ 包含动态时间戳/UUID | 前缀不再稳定 | 每次调用都是新前缀 | 移除 Prompt 中的动态内容 |
| ❌ `/compact` 频繁触发 | compact 使用不同的系统 Prompt | 缓存前缀改变 | 手动控制在必要时 compact |
| ❌ `/resume` 恢复旧会话 | 序列化方式略有不同 | 前缀不匹配 | 新任务开新会话 |

**验证缓存是否生效**：

```bash
# 在 Claude Code 中
/cost

# 查看 usage 字段：
# cache_read_input_tokens 越大越好
# 如果一直为 0 → 缓存前缀被破坏了
```

### 杠杆 2：模型分层路由（节省 20-35%）

```
任务复杂度判断 → 路由到不同模型

┌──────────────────────────────────────┐
│ 简单任务（补全、格式化、单行修改）     │
│ → DeepSeek V4 Flash（$0.14/$0.28）   │
│ → Haiku（订阅内含，不额外计费）       │
├──────────────────────────────────────┤
│ 中等任务（单文件实现、代码审查）       │
│ → Gemini 3.1 Pro（$2/$12）           │
│ → Sonnet（订阅内含）                  │
├──────────────────────────────────────┤
│ 复杂任务（架构设计、多文件重构）       │
│ → Opus（只在必要时用）               │
└──────────────────────────────────────┘
```

**Claude Code 实操**：

```bash
# 切换模型
/model sonnet    # 日常任务
/model opus      # 复杂重构时才切换
/model haiku     # 简单问答
```

### 杠杆 3：CLAUDE.md 精简（节省 10-20%）

CLAUDE.md 是**每次 API 调用的固定前缀**，100% 参与 token 计费。

| CLAUDE.md 大小 | 每次调用开销 | 100 轮对话开销 |
|---------------|------------|---------------|
| 200 行（过度详细） | ~3K token | ~300K token |
| 50 行（精简索引） | ~800 token | ~80K token |
| 差异 | **3.75x** | **节省 $3-8/会话** |

**精简策略**：CLAUDE.md 作为"索引"，详细信息放在单独文件里，让 AI 按需读取。

```markdown
<!-- 精简版（推荐） -->
# Project: Game Server
- Build: cmake -B build && cmake --build build
- Test: cd build && ctest
- Code style: Google C++ Style Guide, see docs/style.md
- Architecture: see docs/architecture.md
- Conventions: C++17, no exceptions, RAII, snake_case
- Constraints: no raw new/delete, all IO must have timeout

<!-- 而非冗长版 -->
# 不推荐在 CLAUDE.md 中写数百行详细的架构文档
# 那些内容应该放在独立文件中，AI 需要时自动读取
```

### 杠杆 4：effortLevel 调节（节省 20-30% 输出）

```json
// ~/.claude/settings.json
{
  "effortLevel": "medium"
}
```

| effortLevel | 输出风格 | 适用场景 |
|-------------|---------|---------|
| `low` | 简洁答案，不过度解释 | 简单查询、格式化 |
| `medium` | 平衡（推荐日常） | 日常编码 |
| `high` | 深度思考、详细解释 | 复杂架构设计 |

### 杠杆 5：会话生命周期管理

```
✅ 好习惯：
- 一次会话聚焦一个任务
- 任务完成后 /clear 或开新会话
- 长任务中途主动 /compact
- 休息 >1 小时 → 开新会话（Cache TTL 到期了）

❌ 坏习惯：
- 一个会话做 10 件不相关的事
- 从不 /compact，上下文撑爆
- /resume 恢复 2 天前的会话
- 在同一个会话中反复切换模型
```

## 🛠️ 三、C++ 专属省钱技巧

### 技巧 1：选择性提供上下文

```
❌ 浪费：
"帮我看看这个项目" → AI 会串行读取所有文件

✅ 节省：
"帮我看看 src/network/ 下的事件循环代码，
  只看 event_loop.cpp 和 connection.cpp"
```

### 技巧 2：先让 AI 搜索，再让 AI 读

```
❌ 浪费：
"这个函数在哪里被调用？" → AI 可能读很多无关文件

✅ 节省：
"用 grep 搜索 handleRequest 的所有调用位置，
  然后只读取前 5 个结果"
```

### 技巧 3：编译错误精简后再喂给 AI

```
❌ 浪费：
直接把 500 行模板错误粘贴给 AI → 大量 token 花在噪音上

✅ 节省：
先用 `2>&1 | head -50` 截取关键错误
再用 `-Wfatal-errors` 只拿第一个错误
```

### 技巧 4：利用 Fork Agent 的 Cache 共享

```
✅ 并行子 Agent（Workflow）共享父会话的 Prompt Cache
→ 创建 5 个子 Agent 并行分析代码
→ 输入 token 成本 ≈ 1 个 Agent 的成本（~95% 缓存命中）
```

## 📊 四、成本监控

### Claude Code 内置

```bash
/cost                    # 当前会话 token 消耗
```

### API 直连监控

```bash
# 设置预算告警
export ANTHROPIC_MAX_COST_PER_SESSION=5.00

# 或在 settings.json 中
{
  "maxCostPerSession": 5.00
}
```

### 月度成本估算

| 使用强度 | 日均 Token | 月成本（Max 订阅） | 月成本（API 直连） |
|---------|-----------|-------------------|-------------------|
| 轻度 | ~100K | $20（Pro） | $15-30 |
| 中度 | ~500K | $100（Max 5x） | $75-150 |
| 重度 | ~2M | $200（Max 20x） | $300-600 |
| 超重度 | ~10M | $200（可能限速） | $1,500+ |

> 💡 **结论**：重度使用选 Max 订阅制（$100-200/月），远低于 API 按量付费（$600-1,500/月）。
